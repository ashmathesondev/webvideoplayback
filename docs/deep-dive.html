Architecture & Implementation

Inside the Video Player

A detailed walkthrough of how every subsystem works — backend selection, threading, synchronization, the decode pipeline, HTTP streaming, and performance measurement. Written for someone who knows some programming but is new to multimedia systems.

Decoder Backend Model

The player loop talks to decoder and render interfaces. The selected decoder backend owns the details of demuxing, decoding, timing, and audio queueing.

Main Thread
Runs the SDL event loop. Handles keyboard/mouse input. Pulls decoded video frames from the queue and renders them to the GPU. Responsible for everything visible on screen.
FFmpeg Backend
The default path. It uses FFmpeg to open containers, decode video on a worker, feed audio packets to an audio worker, and present frames through SDL textures.
Media Foundation Backend
The opt-in Windows native path. It uses Media Foundation Source Reader output for video and audio, then queues BGRA frames and float PCM through the existing SDL layers.
Backend selection: auto currently selects FFmpeg. native selects Media Foundation on Windows. The active backend appears in the window title.
Bounded Packet Queues

Backends communicate with the render loop through bounded queues or bounded read-ahead. When a queue is full, the producer waits instead of filling all available memory.

  • FFmpeg video frame queue: decoded frames waiting to be rendered
  • FFmpeg audio packet queue: compressed packets waiting for audio decode
  • Media Foundation video queue: frames read while filling audio
  • Blocking prevents unbounded memory growth if decoding outruns rendering
  • Implemented with std::mutex + std::condition_variable (C++ standard library)
PlaybackPause — Shared Atomic Gate

When you drag or resize the SDL window, the OS can steal time from the render thread. If decode keeps running, frames pile up and audio gets ahead. PlaybackPause gates backend work during window interaction.

  • Implemented as an std::atomic<bool> shared across playback components
  • Atomics are thread-safe without a mutex — any thread can read/write simultaneously
  • Decode paths check the flag before each unit of work
  • Prevents audio "galloping ahead" of video during window moves
Audio as the Master Clock — Why Audio Wins

Video and audio are separate streams in a container file. After decoding they need to stay in sync — if video runs fast, you see frames from the future while hearing the past (lip-sync problems). The industry-standard solution: make audio the master clock and let video chase it.

Audio output is driven by hardware — the sound card requests samples at a fixed rate. This rate is extremely stable, making the audio playback position the most reliable clock in the system. Video rendering is flexible (you can drop a frame), so video waits for audio.

The sync rule: Before rendering a frame, check if target_seconds <= audio_playback_seconds + 0.005. If audio hasn't caught up yet, sleep briefly and check again. If the frame is more than 250 ms behind audio, drop it — it's stale.
Sync Logic (pseudocode)
// In the main render loop, for each video frame:
double target = frame_seconds(frame);
double audio  = audio_output.playback_seconds();

if (target + 0.250 < audio) {
    // Frame is 250ms behind — drop it
    drop_frame();
    continue;
}

while (target > audio + 0.005) {
    // Too early — wait for audio to catch up
    SDL_Delay(1);
    audio = audio_output.playback_seconds();
}

// Audio has caught up — render this frame
video_output.render_frame(frame);

What is a container vs. a codec?

Think of a container (.mp4, .mkv) as an envelope. It holds the actual compressed data plus metadata (timecodes, chapter markers, subtitle tracks). The codec (H.264, VP9, AV1, AAC) is the specific compression algorithm used to shrink the raw data. You can have H.264 video inside either an MP4 or an MKV container — the envelope and the compression are independent.

What is YUV420P and why is it a "fast path"?

YUV420P is how most video codecs (H.264, H.265) store decoded frames: brightness (Y) at full resolution, color (U and V channels) at half resolution. SDL has a native call SDL_UpdateYUVTexture that accepts YUV420P directly — the GPU can display it without conversion. For other pixel formats, libswscale must first convert to RGBA, which costs extra CPU time every frame.

RAII Wrappers — Memory Safety Around C APIs

FFmpeg and Media Foundation are C-style APIs. Handles must be released on every success and error path.

The FFmpeg path uses RAII wrappers in src/player/av_support.hpp. The Media Foundation path uses WRL ComPtr wrappers for COM interfaces.

RAII principle: Acquire a resource when you construct an object; release it when the object is destroyed. The destructor runs even if an exception throws, so you can never forget to clean up. 
// Defined in av_support.hpp:
using FormatPtr = std::unique_ptr<
    AVFormatContext,
    FormatDeleter  // calls avformat_close_input
>;

using CodecContextPtr = std::unique_ptr<
    AVCodecContext,
    CodecContextDeleter  // calls avcodec_free_context
>;

using FramePtr = std::unique_ptr<
    AVFrame,
    FrameDeleter  // calls av_frame_free
>;

// Usage — no manual cleanup needed:
FormatPtr fmt = open_input(url);
// fmt automatically freed when it goes out of scope
AudioOutput — How Sound Gets Played

Defined in src/player/audio_output.hpp. AudioOutput pushes decoded float samples into SDL's internal audio queue and tracks the playback clock.

  • Target audio buffer size: 150 ms (tunable via WEBVIDEOPLAYBACK_AUDIO_LATENCY_MS env var)
  • FFmpeg pipeline: AVFrame → libswresample → float32 stereo → SDL stream
  • Media Foundation pipeline: float PCM sample → SDL stream
  • playback_seconds() returns a running estimate of how much audio has been consumed by the hardware
  • This estimate is the master clock that video synchronizes against
  • Higher latency = more stable audio (fewer underruns) but more sync lag
Render Sink — Letterboxing and GPU Rendering

Handles window creation, GPU texture management, the backend-labeled title bar, and letterboxing. The calculation computes a video_destination_rect that preserves the video's original proportions.

  • SDL_Renderer uses the GPU for texture scaling — no CPU-side pixel scaling
  • YUV420P fast path: calls SDL_UpdateYUVTexture (three-plane Y/U/V upload)
  • YUY2 and BGRA fast paths: upload directly with SDL_UpdateTexture; other pixel formats convert to RGBA first
  • Letterbox bars cleared to black each frame before rendering the video rect
  • F1 key toggles debug overlay drawn with a custom bitmap font glyph table
F1 Debug Overlay

Press F1 to show a real-time stats panel rendered directly on top of the video. The overlay is drawn using a hardcoded bitmap font — a table of 8×8 pixel glyphs for every printable ASCII character. This avoids any font library dependency for the HUD.

  • Renders: FPS, frame decode time (ms), GPU upload time, render time, audio clock drift
  • Memory stats: SDL texture VRAM estimate, decoded frame queue depth
  • Pixel-drawn using SDL_RenderDrawPoint — no texture atlas or font library needed
Performance CSV Reports

Pass --performance-report on startup to record per-frame timing to a CSV file in the reports/ directory. Every frame writes one row.

  • Columns: frame number, pts (seconds), demux_ms, decode_ms, swscale_ms, upload_ms, render_ms, audio_drift_ms
  • Use any spreadsheet or Python pandas to plot and analyze bottlenecks
The HTTP Test Server — How Streaming Works

Real streaming services (YouTube, Netflix) don't send you the entire video file before you can watch it. They send it piece by piece using HTTP byte-range requests — your player asks "give me bytes 1,000,000 to 2,000,000", downloads that chunk, plays it, then asks for the next chunk. This lets you seek to any position without re-downloading the whole file.

The test server in this project implements exactly this. It serves files from disk over HTTP and understands the Range: bytes=X-Y header. FFmpeg and Media Foundation both use byte-range requests when opening HTTP media.

Security note: resolve_request_path() blocks path traversal attacks (/../ sequences in URLs). It normalizes the path and verifies the resolved file is inside the configured root directory before opening it.
Server Threading
Accept Loop Thread
Blocks on accept() waiting for new TCP connections. Spawns a new handle_client thread for each connection.
handle_client Threads
One thread per connection. Parses the HTTP request, resolves the path, calls send_file() with byte ranges, closes socket when done.
Dashboard Refresh Thread
Wakes every 100 ms, updates FTXUI component state with latest transfer stats, triggers a screen re-draw.
FTXUI Event Loop
Runs the terminal UI event loop. Handles keyboard input (Q to quit). Draws the transfer progress table using FTXUI components.

Key Design Decisions and Why
Audio latency env var instead of a flag
Audio buffer tuning is system-specific (some machines need bigger buffers to avoid pops). An environment variable lets you set it once for your machine without changing command lines or rebuilding.
WIN32 subsystem flag on the player
CMakeLists sets WIN32 for the player target. This starts the process without a console window (no black box behind the video). SDL's preprocessor macro translates main into the Windows WinMain entry point automatically.
Per-connection threads in the server
The test server spawns one OS thread per incoming connection. Simple to reason about, no async complexity. Fine for a dev-only tool where concurrency stays low. Not appropriate for production (too many threads under load).
Custom bitmap font for the debug overlay
Using a real font library for an optional 8×8 debug HUD would add a significant dependency. A hand-coded glyph table (an array of uint8_t bitmasks per character) is self-contained, always available, and renders at exactly one pixel per bit.
250ms stale-frame threshold
If video decoding stalls and then catches up, old frames would flash briefly before syncing. Dropping frames older than 250 ms relative to audio ensures the visual experience stays smooth even after a decoder hiccup.
Reports directory set at compile time
CMake defines WEBVIDEOPLAYBACK_REPORT_DIR as an absolute path at build time. This means performance CSVs always land in the source tree's reports/ folder, making them easy to find for analysis regardless of where you launch the binary from.
