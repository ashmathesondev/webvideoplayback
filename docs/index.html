Project Overview

How does a video player actually work?

This C++ project builds a real media player from scratch — showing every step from opening a video file to rendering pixels on screen and playing audio through your speakers. It also includes a local HTTP streaming server for testing network playback.

two separate programs with different jobs
The Two Programs
webvideoplayback.exe — The Player

The main media player. Think of it like VLC or Windows Media Player, but built from scratch in this codebase so you can read every line of code that makes it work. It opens video and audio files, decodes them, and plays them back in a resizable window.

  • Launch with no arguments — a native Windows file picker opens
  • Pass a file path or URL as an argument to skip the picker
  • Press F1 to show a real-time debug overlay
  • Press ESC to close the player
  • Uses --performance-report flag to record per-frame timing CSV files
webvideoplayback_test_server.exe — The Streaming Server

A lightweight web server that serves video files over HTTP from your local machine. This lets you test how the player handles streaming over a network connection — the same way a website like YouTube streams video to your browser.

  • Binds to localhost:8080 by default (configurable)
  • Supports HTTP byte-range requests so the player can seek and resume
  • Shows a live terminal dashboard with active transfer progress bars
  • Configurable via a JSON file listing which files to serve and at which URL paths
  • Press Q in the terminal to stop the server
four libraries doing the heavy lifting
The Four Libraries — What Each One Does
FFmpeg
The Codec and Decoder Engine

FFmpeg understands hundreds of video and audio formats — MP4, WebM, MKV, MP3, AAC, Opus, and many more. When you have a compressed video file, FFmpeg knows how to decompress it into raw pixel data (for the screen) and raw audio samples (for the speakers). Writing all those decoders from scratch would take years; FFmpeg provides them ready to use.

SDL2
The Display and Sound Output Layer

SDL2 (Simple DirectMedia Layer) handles everything you actually see and hear: it creates the window, uses the GPU to draw pixels to the screen as fast as possible, and talks to the Windows audio system to play sound. Without SDL2, you would need to write different code for every operating system just to open a window or play a sound.

FTXUI
The Terminal Dashboard Library

FTXUI builds text-based user interfaces that run inside a terminal window — the kind with boxes, progress bars, and tables made of characters. The test server uses FTXUI to show a live dashboard of which files are being streamed, how much data has been sent, and how long each transfer took.

nlohmann-json
The JSON Config Parser

A widely used C++ library for reading and writing JSON files. The test server uses it to load a config file that maps URL routes (like /movie.mp4) to actual file paths on disk. This lets you control which files the server serves without recompiling the code.

building and running the project
Build and Run
Build (PowerShell)

Uses CMake with vcpkg. All four libraries download and build automatically — no manual installs needed.

cmake --preset windows-msvc-vcpkg
cmake --build --preset windows-msvc-vcpkg-debug
Run the Player

No arguments opens a file picker. Pass a URL or path to skip it.

.\build\...\webvideoplayback.exe

# Play from test server:
.\build\...\webvideoplayback.exe `
  http://127.0.0.1:8080/video.mp4
Run the Test Server

Point it at a folder of media files. Opens a live dashboard in the terminal.

.\build\...\webvideoplayback_test_server.exe `
  --root E:\path\to\media `
  --port 8080
Useful Flags and Settings
  • --performance-report — save per-frame timing CSV to reports/
  • --config file.json — server loads explicit file routes from JSON
  • WEBVIDEOPLAYBACK_AUDIO_LATENCY_MS — env var to tune audio buffer size (default 150 ms)
  • Release build path: build/.../Release/webvideoplayback.exe
What makes this project special: Unlike most media players, this one deliberately exposes every internal timing measurement through the debug overlay (press F1) and optional CSV performance reports. You can see exactly how many milliseconds demuxing, decoding, pixel conversion, GPU upload, and rendering each take per frame — plus watch how audio and video stay synchronized in real time.
